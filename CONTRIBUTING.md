# Contributing Guide for Design System 2 Starter Kit

Thanks for your interest in contributing! This page covers the governance model, how to report issues, and how to submit changes.

# Governance Model

## Salesforce Reviewed

This project is a *community-supported* initiative and does not have a formal maintenance SLA. While we actively encourage involvement, please keep the following guidelines in mind:
- **Review Process:** All contributions are reviewed by designers or engineers within the Salesforce UX team.
- **Decision Making:** Salesforce maintainers serve as the final arbiters on all roadmap decisions and pull request approvals.

# Issues, requests & ideas

Use the GitHub Issues page to submit issues, enhancement requests, and discuss ideas.

### Bug Reports and Fixes
- If you find a bug, search for it in [Issues](https://github.com/salesforce-ux/design-system-2-starter-kit/issues). If it isn't already tracked, [create a new issue](https://github.com/salesforce-ux/design-system-2-starter-kit/issues/new).
- Issues identified as a bug will be labelled `bug`.
- If you'd like to submit a fix, [send a Pull Request](#creating-a-pull-request) and mention the Issue number.

### New Features
- Describe the problem you want to solve in a [new Issue](https://github.com/salesforce-ux/design-system-2-starter-kit/issues/new).
- Issues identified as feature requests will be labelled `enhancement`.
- Please wait for feedback from maintainers before spending too much time writing code — some enhancements may not align with current project objectives.

### Tests, Documentation, Miscellaneous
- If it's a trivial change, go ahead and [send a Pull Request](#creating-a-pull-request).
- If not, [open an Issue](https://github.com/salesforce-ux/design-system-2-starter-kit/issues/new) to discuss the idea first.

# Contribution Checklist

- [x] Clean, simple, well styled code
- [x] Commits should be atomic and messages must be descriptive. Related issues should be mentioned by Issue number.
- [x] Dependencies — minimize number of dependencies; prefer Apache 2.0, BSD3, MIT, ISC and MPL licenses.
- [x] Changes must be approved via peer code review.

# Creating a Pull Request

1. **Ensure the bug/feature was not already reported** by searching on GitHub under Issues.
2. **Fork** the repo and **clone** your fork locally.
3. **Create** a new branch (e.g. `git checkout -b fix-issue-11`).
4. **Commit** changes to your branch.
5. **Push** your branch to your fork (e.g. `git push origin fix-issue-11`).
6. **Submit** a Pull Request against the `main` branch and reference the issue(s) you are fixing. Keep it simple and small.
7. **Sign** the Salesforce CLA (you will be prompted to do so when submitting the Pull Request).

> **NOTE**: Be sure to [sync your fork](https://help.github.com/articles/syncing-a-fork/) before making a pull request.
